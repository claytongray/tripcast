# Epic 7 Retrospective — Admin Observability Panel

**Date:** 2026-07-01 · **Facilitator:** Amelia (Developer) · **Project Lead:** Clayton
**Status:** First retrospective for the project (no priors to carry forward).

## Epic summary

- **Stories:** 7/7 done (7.1–7.7); 7.8 (admin demo seeder) deferred to backlog.
- **Scope:** Phone-first, read-only admin panel under the single `admin` Gate (AD-12) — six sections (Overview, Users, Emails, Promos, Samples + folded-in Monitoring) on a shared metrics/charting foundation. Plus a post-implementation "projected sends" enhancement in the Emails section.
- **Quality:** 361 tests / 1498 assertions; pint/phpstan/types/lint/ssr green. 0 migrations (schema untouched). 0 production incidents (not yet deployed).
- **Commits:** `f291cd7` panel (7.1–7.7) · `a06b82c` code-review fixes (9) · `5b89fd9` projection.

## What went well

- **Foundation-first compounded.** 7.2 (`MetricsService`, `MetricsWindow`, `KpiTile`, `TrendChart`) was built once and consumed by 7.3–7.7 via an identical section-builder + controller-window + window-selector pattern. Each subsequent section was faster and more consistent.
- **Low-risk by design.** Reusing the single `admin` Gate and staying strictly read-only meant no new auth surface and no possibility of data mutation across the whole epic.
- **Reused authorities, not duplicates.** The projection extended `CadencePredicate` (AD-11) via a shared `dueQuery()` (`dueOn`/`dueCountOn`); liveness reused the existing `digests:run` signal (cache snapshot, no new table).
- **Test discipline held.** Every story shipped with feature tests, including a no-N+1 assertion (7.4) and a success-rate regression (review fix) that protect real behavior.

## Challenges / recurring patterns

- **PHPStan friction with dynamic query aliases** — `withCount`/`withMax`/`withExists`/`selectRaw ... as aggregate` recurred in 7.4, 7.6, 7.7; same fix class each time (`getAttribute`, union-literal `@param` types, `foreach` to build a `list`). Re-derived rather than documented.
- **Whole-number float serialization** (`50.0 → 50`, decodes as int) tripped assertions in 7.3, 7.5, 7.6 — a repeated test-writing gotcha (fix: numeric closure `fn ($v) => (float) $v === 50.0`).
- **Green gates ≠ correct.** Adversarial code review caught 5 real bugs the passing test suite missed: success-rate counting in-progress `sending` rows (and diverging from the Emails page), an unregistered Chart.js `Legend`, and a CTR sparkline fed raw click counts. No test exercised those paths.
- **Environment drift.** `/admin/overview` 500'd in dev because the `sample_requests` migration was pending — the test suite passed because `RefreshDatabase` rebuilds the test DB. "Tests green" masked "dev DB stale."
- **Duplication accrued** building sections fast (`counts()` helper ×3, window-switcher nav ×4, days-param block ×4) — all flagged in review and cleaned, but a mid-epic consolidation pass would have pre-empted it.
- **Deferring 7.8** leaves the panel sparse locally, which also delayed noticing the migration drift.

## Significant discovery — impacts Epic 8

7.6's promo analytics derives "weather profile" by **inverting the static `config('tripcast.promo.catalog')`**. Epic 8 makes the catalog **DB-backed (`PromoItem`)** and introduces the **first mutations** (CRUD) and a `DatabasePromoProvider`. Therefore:
- The slug→profile inversion must move to the DB source (group by `PromoItem.weather_profile`).
- Epic 8 breaks Epic 7's read-only/single-Gate simplicity — the catalog write-auth model needs a decision at planning time.
