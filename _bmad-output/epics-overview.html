Tripcast — what's being built, by epic

Five epics that take a traveler from a landing-page form to a calm daily weather digest — passwordless, reliable, and (eventually) self-sustaining on affiliate revenue. Build in order; each epic stands alone but enables the next.

Epic 2

The Daily Digest

core product in progress

An active trip gets the correct morning email each day from forecast-window open (departure − 7d) through the return date, then stops — a faithful 7-day forecast in °F + °C, countdown copy, login-free footer actions, and one-tap feedback. The send is reliable, idempotent, and self-monitoring.

2.1review

Weather provider port

A swappable WeatherProvider that fetches a fresh 7-day forecast by coordinates; partial data → a "limited" marker, never fabricated values.

2.2review

Cadence predicate + daily command

One authority for "is a digest due today," and a scheduled command that dispatches one job per due trip — the scaling seam.

2.3review

Per-trip job: claim-first + snapshot

Each job claims its (trip, date) row before any work (unique index = no double-send), fetches once, and persists the snapshot before delivery.

2.4review

Render + bounded-retry delivery

Render the digest from the persisted snapshot; retry delivery ≤3× in-process, then a terminal sent/failed. Plain-text twin; legible with images blocked.

2.5review

End-trip / unsubscribe links

Signed, trip-scoped footer links: GET confirms, POST changes state. End-trip completes one trip; unsubscribe suppresses all of the user's.

2.6backlog

Feedback click

One-tap 👍 / 👎 in the footer, no login, tied to trip + send date; re-tap idempotent, collapses to "Thanks — noted."

2.7backlog

Daily-run liveness heartbeat

Every run records its outcome + a heartbeat; a missed beat or finished-zero-when-due fires an out-of-band alert to the builder.

Epic 4

Forecast History & AI Narration

backlog

Each daily fetch is retained as a day-by-day forecast-history series (over email_logs, purged ~30 days after return), and from it the digest gains a short, calm, AI-generated line about notable day-over-day changes — enhancement-only, never blocking or delaying a send. The product's first AI feature.

FR-15 FR-16  ·  anchored by AD-9 EmailLog series · AD-16 bounded-retention sweep · AD-17 Narrator port off the delivery path

4.1backlog

Forecast-history series + purge

The per-send snapshots are the history (no new store); a daily sweep nulls only weather_snapshot ~30 days after return, leaving the outcome row.

4.2backlog

Narrator port — day-over-day line

A ClaudeNarrator writes one never-alarmist line grounded only in stored snapshots; no prior snapshot / timeout / error → line omitted, digest still sends.

Epic 5

Affiliate Monetization

the pivot backlog

Free-tier digests carry one calm, native, weather-keyed Amazon recommendation below the forecast (with mandatory disclosure), suppressed for ad_free users; clicks route through a signed redirect that attributes engagement before forwarding to Amazon. Built against a placeholder catalog so nothing waits on curated content.

FR-17 FR-18  ·  anchored by AD-18 PromoProvider port (render-slot only) · AD-19 plan entitlement predicate · AD-6 signed promo redirect

5.1backlog

Entitlement predicate

One shouldShowPromo(user) derived from users.plan (free → on, ad_free → off); no other call site re-implements the check.

5.2backlog

PromoProvider + placeholder catalog

A port reading a config catalog of weather-profile → product entries; selects one via deterministic send_date rotation, generic fallback on no match.

5.3backlog

Promo slot + disclosure

One native unit below the forecast with the mandatory Amazon-Associate disclosure (HTML + text); ad_free / error / timeout → slot simply absent.

5.4backlog

Signed-redirect attribution

Impression logged at render; a tap reads-then-logs a click to promo_events (idempotent) then forwards to the Amazon URL — prefetch-safe, no PII.

Browser testing — end of Epic 2 & 3

The Pest suite already covers the happy path. What you do by hand is the stuff tests can't see: cross-client email rendering, real interaction states, and the error / edge branches (UX-DR15). Epic 2 is mostly email rendered in a browser + the login-free link pages; Epic 3 is the web app itself.

The digest only renders when a job runs. Seed a due trip (confirmed user, active, departure within 7 days) and run the daily command. Point mail at Mailtrap (not the log driver) so you can open the HTML in a browser and forward to real clients. Shift the trip's departure / return dates and re-run to hit each countdown boundary.
