# Epic 11 Retrospective — Weather Provider: Apple WeatherKit Migration

**Date:** 2026-07-04
**Facilitator:** Amelia (Developer) · **Project Lead:** Clayton

## Epic summary

Replaced the WeatherAPI.com forecast provider with Apple WeatherKit behind the existing `WeatherProvider` port, to fix a 5–8°F high-temperature inflation on hot/humid inland days (WeatherAPI was returning a heat-index-like figure as the "high"), and render the Apple Weather attribution its license mandates.

| Story | Outcome |
| --- | --- |
| 11.1 — WeatherKit provider behind the port | `WeatherKitProvider` + cached ES256 JWT + `ConditionCode`/emoji mapping. Two adversarial review rounds (Opus, then cross-model Fable 5) → 15 patches incl. 2 HIGH-class fixes (exception wrapping, `resolve()` cache-escape). |
| 11.2 — Destination timezone capture at trip creation | `trips.destination_timezone` resolved from coordinates (Google Time Zone API) at creation; `trips:backfill-timezones` for existing trips. Cross-model review. |
| 11.3 — Apple Weather attribution + cutover | Inlined, provider-gated, dark-mode-legible mark in digest + sample footers & text twins; `weatherkit:check` cutover preflight; hard cutover via env flag. |

**Delivery:** 3/3 stories done. 617/617 tests green; pint, phpstan, eslint, vue-tsc, build:ssr all clean.

**Production outcome (validated by Clayton):** `weatherkit:check --live` PASS; live Apple fetch **Kennett 97°F** (realistic air temp, not the 105°F heat-index). `TRIPCAST_WEATHER_PROVIDER=weatherkit` flipped; `.p8` secured (0600) in shared `storage/app/private/` (persists across deploys); `trips:backfill-timezones` filled **14/14** active-trip zones — including Europe/London, Europe/Madrid, Europe/Berlin, America/Chicago that were previously wrong under the ET fallback.

**Incidents:** 1 — an SSR daemon crash inherited from an earlier GA4 commit (not Epic 11 code), surfaced by the Epic 11 deploy. Root-caused, fixed, and regression-tested.

## What went well

1. **Port/adapter + config flag turned a scary migration into a safe one.** A completely new API — different auth (ES256 JWT), different units (°C/decimals), a different response shape — landed inside a single adapter behind `WeatherProvider`. Nothing upstream knew the provider changed. It shipped *dark* (default `weatherapi`) and cut over with a single env flag, fully reversible. **Clayton's read: "wasn't too bad to integrate a completely new API"** — the abstraction is why.
2. **The business goal is provably fixed in prod.** Accurate highs (97°F, not 105°F) confirmed by a live fetch, not just tests.
3. **Cross-model adversarial review earned its cost.** Reviewing 11.1 with Opus *and* Fable 5 caught 2 HIGH-class bugs a single pass would likely have missed.
4. **Ops preflight thinking caught a deploy trap before it bit.** The git-ignored `.p8` would not survive a zero-downtime deploy inside the release tree; this was found by reasoning about Forge mechanics and building `weatherkit:check`, rather than discovering it via a silently-failed digest send.

## What was rough

1. **A shared client+SSR entry is a silent-failure trap.** `initializeAnalytics()` (from the earlier GA4 commit) ran in the Node SSR render via `app.ts` and dereferenced `window`/`document` with no guard → SSR bundle threw on every render → SSR daemon crash-loop → prod silently fell back to client-side rendering. There was **no SSR smoke test**, so it stayed invisible until a deploy exposed it.
2. **The deploy pipeline conflates "SSR restart hiccup" with "deploy failed."** The release activated fine; only the `inertia:stop-ssr` nicety errored (it couldn't reach a crash-looping daemon), yet Forge reported the whole deploy failed.
3. **Live-data ACs drift.** 11.3's AC hard-coded "97°F/86°F," but WeatherKit is a live forecast — the numbers change daily, so the tests had to assert *substance* (no heat-index inflation, realistic range), not exact values.

> **Decision (2026-07-04):** No deploy-pipeline SSR gates. A live SSR render inside the deploy script collides with the running daemon (hardcoded port 13714), and `inertia:stop-ssr`'s exit code is an ambiguous signal anyway. SSR is instead verified **locally before push** with `npm run build:ssr && php artisan test --filter=SsrRender` (and CI runs the same `SsrRenderTest` when triggered). The `|| true` / `check-ssr` deploy additions were considered and declined as over-engineering for a solo workflow.

## Key takeaways

1. A port/adapter + config flag turns a scary provider migration into a dark-shipped, env-flip cutover with instant rollback.
2. Cross-model adversarial review is worth the cost — it catches HIGH-class bugs a single pass misses.
3. Shared client+SSR entries are a silent-failure trap; the fix is cheap, but you need a test that *runs* the SSR bundle to see it — and it belongs in CI, not the deploy script.
