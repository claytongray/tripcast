# Epic 8 Retrospective — Sponsored Catalog & Weather Mapping

**Date:** 2026-07-01
**Participants:** Clayton (Project Lead), Amelia (Developer), Winston (Architect), John (PM)
**Status:** All 5 stories done and code-reviewed on branch `epic-8-sponsored-catalog` (not yet merged/deployed).

## Epic summary

Turned the static, config-based weather-keyed promo catalog into an admin-managed, DB-backed system behind the existing `PromoProvider` port, with a date-ranged Featured override and an Essentials fallback — without changing `promo_events` or the deterministic `send_date` rotation.

- **Stories:** 8.1 `promo_items` foundation → 8.2 `DatabasePromoProvider` (+ safe switchover) → 8.3 catalog CRUD UI → 8.4 `mild` re-bucket + precedence guards → 8.5 per-item performance + analytics repoint.
- **Tests:** 388 → **429** (+41 over the epic); every story shipped gates-green (Pint / PHPStan / types / lint / SSR).
- **Review:** 3 adversarial layers (Blind Hunter, Edge Case Hunter, Acceptance Auditor); **0** violations on the binding constraints (AD-12 Gate, AD-18 attribution, byte-identical determinism, `mild`→Essentials, canonical columns, 8.5 repoint). 4 patches applied, 1 deferred, 4 dismissed.

## What went well

- **The port/adapter seam paid off.** `DatabasePromoProvider` slid behind the existing `PromoProvider` interface with byte-identical `crc32(send_date) % count` rotation **and** an empty-table → config fallback, so the provider swap kept every legacy Promo/Digest test green **unchanged**. The empty-table fallback is the reusable pattern of the epic — a zero-risk adapter switchover.
- **Clean story slicing.** Each story was independently shippable with a never-blank safety net; `slug`-as-immutable-attribution-key held across all five (set-once on update, `withTrashed` resolution) — no attribution breakage.
- **Epic 7 prep converted directly into delivery.** The two open Epic 7 questions — catalog write-auth model and moving slug→profile into the DB — were resolved as the core of Epic 8 (single admin Gate for writes; analytics repointed to `PromoItem`).
- **Adversarial review came back clean on the hard stuff.** No auth / mass-assignment / injection / determinism / attribution defects across three independent layers.

## Challenges / recurring patterns

- **A cross-story binding decision slipped between stories.** The epic header said "8.2 re-buckets the `mild` item," but 8.2 left the seeder untouched; the re-bucket actually landed in 8.4, caught while scoping it. It lived only in the header, never as a task on a specific story.
- **Validation ↔ persistence drift** (two review patches): `image_url`/`url` columns were `VARCHAR(255)` while validation allowed `max:2048`; `featured_from` wasn't `required_with:featured_to`. Same class — the FormRequest disagreed with the schema/scope, and the 255 mismatch had sat since 8.1's migration.
- **Business rule enforced in the UI, not the request.** "No new `mild` items" was enforced only by the create form omitting the option; a crafted POST could still create an unreachable row (fixed in review with a create-aware rule).
- **Flaky test from unpinned factory state.** The `featured()` factory left `weather_profile` random, making a determinism test non-deterministic (surfaced during 8.3's full-suite run, fixed there).
- **Placeholder-vs-real-product ambiguity.** Real reasoning cycles went into the `packing-cubes` re-bucket before clarifying those are placeholder scaffolding, not real products — which set the demo-fallback-vs-real-product strategy (placeholders stay as dev/demo fallback; prod gets real products via the CRUD).
